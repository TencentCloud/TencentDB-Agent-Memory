# 03 — 一条消息的旅程：数据流全链路

> 阅读时间：~25 分钟 | 建议看完 01、02 再来看

---

## 前言

这一篇我们跟着一条真实消息走完全程。假设用户说：

> **"我平时用 TypeScript，不喜欢 Python"**

来看看这 11 个字经过了哪些处理，最终变成可检索的记忆。

---

## 总体流程图

```
用户: "我平时用 TypeScript，不喜欢 Python"
          │
          ▼
  ┌──────────────────────┐
  │  阶段 1: 自动录制 L0  │  每次对话必然触发
  │  把消息原封不动保存     │
  └──────────┬───────────┘
             │
             ▼
  ┌──────────────────────┐
  │  阶段 2: 触发管道检查  │  判断：现在该不该跑 LLM？
  │  轮次计数 / 空闲超时    │
  └──────────┬───────────┘
             │  条件满足
             ▼
  ┌──────────────────────┐
  │  阶段 3: L1 提取      │  让 LLM 看对话，提取事实
  │  输出结构化记忆         │
  └──────────┬───────────┘
             │  条件满足
             ▼
  ┌──────────────────────┐
  │  阶段 4: L2 场景归纳   │  把零散事实组成场景
  │  输出 Markdown 文件     │
  └──────────┬───────────┘
             │  条件满足
             ▼
  ┌──────────────────────┐
  │  阶段 5: L3 画像更新   │  更新用户画像
  │  输出 persona.md       │
  └──────────┬───────────┘
             │
             ▼
      下次对话时触发召回
```

## 阶段 1: L0 自动录制

**代码位置**：[src/core/hooks/auto-capture.ts](../src/core/hooks/auto-capture.ts)

### 发生了什么

每次 Agent 回复完之后，`index.ts` 里注册的 `agent_end` 钩子会被 OpenClaw 调用：

```typescript
// index.ts（简化）
api.on("agent_end", async (turn) => {
  await core.handleTurnCommitted(turn);
});
```

`handleTurnCommitted` 干了两件事：
1. **存对话**：调 `l0-recorder.ts` 把消息写入 JSONL
2. **通知管道**：告诉 pipeline-manager "有新对话了"

### 实际存在哪

```bash
~/.openclaw/state/memory-tdai/
└── conversations/
    └── 2026-06-30.jsonl    ← 当天的所有对话
```

JSONL 里每一行大概长这样：
```json
{
  "sessionKey": "wechat:user_12345",
  "role": "user",
  "content": "我平时用 TypeScript，不喜欢 Python",
  "timestamp": 1719753600000
}
```

### 同时也存进向量库

除了 JSONL，`auto-capture.ts` 还会把这条消息存到 SQLite 的 `l0_conversations` 表里，顺便用 embedding 算一个向量存进 `l0_vec` 表。这样以后可以搜到原始对话。

## 阶段 2: 管道触发检查

**代码位置**：[src/utils/pipeline-manager.ts](../src/utils/pipeline-manager.ts)

### 不是每次对话都触发 LLM

LLM 调用很贵也很慢，不可能每说一句话就调一次。PipelineManager 用三个条件控制节奏：

```
条件 1: 轮次计数
  每说 N 轮话触发一次（默认 5 轮）
  前几轮有 warmup 机制（第 1 次可能是 1 轮就触发）

条件 2: 空闲超时
  如果用户说了话但不够 N 轮就停了，
  等 600 秒（10 分钟）后自动触发提取

条件 3: 最少间隔
  两次 L1 提取之间至少隔 2 轮，防止频繁触发
```

### 代码片段

```typescript
// pipeline-manager.ts（简化逻辑）
notifyTurnCommitted(sessionKey: string) {
  this.sessionRoundCount.set(sessionKey, count + 1);

  if (count >= this.everyNConversations) {
    // 条件 1 满足：触发 L1 提取
    this.scheduleL1(sessionKey);
  }
  // 同时启动一个 600 秒计时器 → 条件 2
}
```

## 阶段 3: L1 结构化事实提取

**代码位置**：
- [src/core/record/l1-extractor.ts](../src/core/record/l1-extractor.ts)
- [src/core/record/l1-dedup.ts](../src/core/record/l1-dedup.ts)
- [src/core/prompts/l1-extraction.ts](../src/core/prompts/l1-extraction.ts)

### 这一步做什么

管道触发后，`l1-extractor.ts` 调 LLM，把最近的 L0 对话喂给它，让它提取"事实"。

### LLM 看到的输入

```
以下是最近的对话记录：

[user]: 我平时用 TypeScript，不喜欢 Python
[assistant]: 好的，以后我帮你写代码会用 TypeScript

请从以上对话中提取用户的关键信息，以 JSON 格式输出。
```

### LLM 返回的输出

```json
[
  {
    "type": "preference",
    "content": "用户使用 TypeScript 作为主要编程语言",
    "priority": 70
  },
  {
    "type": "preference",
    "content": "用户不喜欢 Python",
    "priority": 50
  }
]
```

### 去重

提取出来的新事实，要和库里已有的比对一下。如果已经有非常相似的记忆，就跳过写入。这就是 `l1-dedup.ts` 干的活：

```typescript
// l1-dedup.ts（简化）
for (const newMemory of extractedMemories) {
  // 用 FTS5 搜索相似的已有记忆
  const ftsQuery = buildFtsQuery(newMemory.content);  // ← 你修过的！
  const similar = await store.searchL1Fts(ftsQuery, 10);

  // 如果相似度超过阈值（embedding 比较），认为已存在
  if (isDuplicate(newMemory, similar)) {
    skip();  // 跳过写入
  }
}
```

### 写入

```typescript
// l1-writer.ts → sqlite.ts upsertL1()
store.upsertL1({
  content: "用户使用 TypeScript 作为主要编程语言",
  type: "preference",
  priority: 70,
  // ... 同时写入向量（embedding）供后续语义搜索
});
```

### 最终存到哪

SQLite 数据库 `vectors.db`：
- `l1_records` 表：存元数据（content, type, priority, timestamps）
- `l1_vec` 表：存向量（float 数组，用于语义搜索）
- `l1_fts` 表：存 FTS5 索引（用于关键词搜索）

## 阶段 4: L2 场景块归纳

**代码位置**：[src/core/scene/scene-extractor.ts](../src/core/scene/scene-extractor.ts)

### 什么时候触发

L2 不会每次 L1 都触发，它有更长的间隔（默认：
- 最少间隔 900 秒 = 15 分钟
- 最长间隔 3600 秒 = 1 小时
- L1 跑完后延迟 10 秒再跑 L2）

### 做什么

调 LLM，把最近新增的 L1 记忆按主题归类，生成人类可读的 Markdown 文件。

### 例子

```
输入（L1 记忆）:
- "用户使用 TypeScript 作为主要编程语言"
- "用户不喜欢 Python"
- "用户偏好使用 VS Code 编辑器"
- "用户习惯用 pnpm 而不是 npm"

LLM 输出（L2 场景）→ 保存为 scene_blocks/dev-preferences.md:
---
# 编程偏好
- 主语言：TypeScript
- 编辑器：VS Code
- 包管理器：pnpm
- 避免：Python
---
```

## 阶段 5: L3 用户画像更新

**代码位置**：[src/core/persona/persona-generator.ts](../src/core/persona/persona-generator.ts)

### 什么时候触发

默认每积累 **50 条新 L1 记忆**才触发一次。画像不需要频繁更新——一个人的偏好不会天天变。

### 做什么

调 LLM，总结所有 L2 场景块，生成一段稳定的人物画像：

```markdown
# persona.md

## 身份特征
用户是一名全栈 TypeScript 开发者，偏好现代工具链。

## 技术偏好
- 编程语言: TypeScript > JavaScript > ... > 不使用 Python
- 编辑器: VS Code
- 包管理: pnpm

## 沟通风格
- 喜欢简洁直接的回复
- 偏好代码示例胜过文字说明
```

### 画像怎么用

每次对话开始时，`auto-recall.ts` 把画像放入 **系统提示词**（不是用户消息），作为 AI 了解你的"背景知识"。

## 阶段 6: 召回——下次对话时

**代码位置**：[src/core/hooks/auto-recall.ts](../src/core/hooks/auto-recall.ts)

### 触发时机

OpenClaw 在每次调 LLM 之前会触发 `before_prompt_build` 钩子：

```typescript
// index.ts（简化）
api.on("before_prompt_build", async (ctx) => {
  const recall = await core.handleBeforeRecall(userText, sessionKey);
  // 把召回的上下文注入到提示词中
});
```

### 召回策略（可配置）

```
strategy: "hybrid"    ← 推荐，向量 + 关键词融合（RRF）
strategy: "keyword"   ← 仅关键词（FTS5 BM25，不需要 embedding）
strategy: "embedding" ← 仅语义搜索（需要 embedding 服务）
```

### hybrid 策略流程

```
用户输入: "帮我写个 TypeScript 函数"
           │
           ├──→ 1. 关键词搜索 (FTS5)
           │     buildFtsQuery("帮我写个 TypeScript 函数")
           │     → FTS5 MATCH 查 l1_fts 表
           │     → 返回 BM25 排名结果
           │
           ├──→ 2. 语义搜索 (Vector)
           │     embedding("帮我写个 TypeScript 函数") → 向量
           │     → vec0 KNN 查 l1_vec 表
           │     → 返回余弦相似度排名结果
           │
           └──→ 3. RRF 融合
                 rrfMerge([fts结果, vec结果])
                 → 综合排名，取 topK
```

### 召回结果注入

```
召回的记忆被格式化成这样，插入到 LLM 的上下文中：

<relevant-memories>
1. [偏好] 用户使用 TypeScript 作为主要编程语言 (置信度: 0.85)
2. [事实] 用户偏好简洁的函数式风格 (置信度: 0.72)
</relevant-memories>

用户消息: 帮我写个 TypeScript 函数
```

AI 看到这个，自然会用 TypeScript 来写，而不是问"你要用哪个语言"。

## 全链路时序总结

```
时刻 T0: 用户说 "我平时用 TypeScript，不喜欢 Python"
时刻 T1: Agent 回复完
时刻 T1+0s:  auto-capture 录制 L0 → JSONL + SQLite
时刻 T1+0s:  pipeline-manager 计数 +1（现在是第 3 轮，不够 5 轮 → 不触发）
时刻 T1+10m: 用户没说话，空闲超时 → 触发 L1
时刻 T1+11m: L1 提取完成，2 条新记忆写入 SQLite
时刻 T1+11m: L2 检测到条件满足 → 触发场景归纳
时刻 T1+13m: L2 完成，"编程偏好"场景生成
时刻 T1+13m: L3 才 2 条新记忆，不够 50 → 不触发
...
时刻 T2: 第二天，用户说 "帮我写个函数"
时刻 T2-Δ:   auto-recall → hybrid 搜索 → 找到 L1 记忆
时刻 T2-Δ:   画像 + 记忆注入 LLM 上下文
时刻 T2:     LLM 回复: "用 TypeScript 实现..."
```
