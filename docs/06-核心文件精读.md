## 文件 1：`index.ts` — 插件的大门

> 源码：[index.ts](../index.ts) | 866 行

### 这个文件干什么

它是整个插件的入口。OpenClaw 启动时调用 `register(api)`，这个函数做四件事：

1. 解析配置
2. 创建 `TdaiCore` 实例
3. 注册两个工具（`tdai_memory_search`、`tdai_conversation_search`）
4. 注册三个钩子（`before_prompt_build`、`agent_end`、`gateway_stop`）

### 关键代码段

#### 入口：`export default function register(api)` (行 132)

```typescript
// index.ts:132
export default function register(api: OpenClawPluginApi) {
```

所有插件都是这样开始的。`api` 是 OpenClaw 给插件的工具箱，你可以通过它注册工具、监听事件、写日志。

#### 创建核心 (行 256-268)

```typescript
// 把 OpenClaw 的 API 包装成统一接口
const hostAdapter = new OpenClawHostAdapter({
  api,
  pluginDataDir,
  openclawConfig: api.config,
});

// 创建核心调度器
const core = new TdaiCore({
  hostAdapter,
  config: cfg,
  sessionFilter,

```

为什么要包一层 adapter？因为同一个 `TdaiCore` 可以跑在 OpenClaw 下，也可以跑在 Hermes HTTP 网关下。adapter 把宿主差异屏蔽了。

#### 注册工具：tdai_memory_search (行 352-435)

```typescript
api.registerTool({
  name: "tdai_memory_search",
```

AI 在对话中说"我要搜索记忆" → OpenClaw 拦截 → 调这个 execute 函数 → 返回搜索结果给 AI。

## 文件 2：`tdai-core.ts` — 总调度器

> 源码：[src/core/tdai-core.ts](../src/core/tdai-core.ts) | 535 行

### 这个文件干什么

它是所有记忆能力的统一入口。不管你调 `searchMemories` 还是 `handleTurnCommitted`，都要通过它。

### 核心结构

```typescript
class TdaiCore {
  // 构造时接收 adapter + config
  constructor(opts: TdaiCoreOptions) { ... }

  // 五个核心方法：
  async handleBeforeRecall()    // 对话前召回记忆
  async handleTurnCommitted()   // 对话后录制
```

### 关键代码段

#### 初始化 (行 143-164)

```typescript
async initialize(): Promise<void> {
  // 1. 创建数据目录 (~/.openclaw/state/memory-tdai/)
  initDataDirectories(this.dataDir);

  // 2. 初始化存储（SQLite 或 TCVDB）
  this.storeReady = this.initStores();

  // 3. 创建管道调度器 + 接线 L1/L2/L3 runner
  if (this.cfg.extraction.enabled) {
    this.scheduler = createPipelineManager(cfg, logger, sessionFilter);
    this.storeReady.then(() => this.wirePipelineRunners());
  }
}
```

注意 `storeReady` 是一个 Promise。后面所有方法都先 `await this.storeReady` 再干活——防止数据库还没初始化好就开始读写。

#### 销毁 (行 169-234)

```typescript
async destroy(): Promise<void> {
  // 销毁顺序非常重要：
  // 1. 等待 store 初始化完成
  // 2. 销毁管道调度器（等正在跑的 L1/L2/L3 完成）
  // 3. 排干后台任务（deferred embedding writes）
  //    ── 如果先关 SQLite，后台任务写不进去就报错
  // 4. 关闭 SQLite 连接
  // 5. 关闭 embedding 服务
}
```

代码里特意标注了一个很长的注释（行 105-123）解释为什么要有 `schedulerStartPromise`——多个 HTTP 请求并发时，必须用 Promise gate 防止重复 start。

#### 召回 (行 244-259)


## 文件 3：`auto-recall.ts` — 记忆召回

> 源码：[src/core/hooks/auto-recall.ts](../src/core/hooks/auto-recall.ts) | 878 行

### 这个文件干什么

**每次 LLM 被调用前，搜出相关记忆，注入到上下文中。**

它是整个记忆系统"对外服务"的出口——记忆存了半天，就是为了这一刻能调出来。

### 总入口：`performAutoRecall` (行 72-100)

```typescript
export async function performAutoRecall(params) {
  // 加了 5 秒超时——如果召回挂了，不能阻塞用户对话
  return Promise.race([
    performAutoRecallInner(params),
    new Promise((resolve) => {
      setTimeout(() => {
        logger.warn("Recall timed out — skipping memory injection");
        resolve(undefined);  // 超时了就不注入记忆，让对话继续
      }, timeoutMs);
    }),
  ]);
}
```

### 核心逻辑：`performAutoRecallInner` (行 102-241)

```typescript
async function performAutoRecallInner(params): Promise<RecallResult> {
  // 步骤 1: 搜 L1 记忆
  const searchResult = await searchMemories(userText, ...);
  let memoryLines = searchResult.lines;

  // 步骤 2: 读 L3 画像文件 → persona.md
  const personaContent = await fs.readFile(pluginDataDir + "/persona.md");

  // 步骤 3: 读 L2 场景导航
  const sceneIndex = await readSceneIndex(pluginDataDir);
  const sceneNavigation = generateSceneNavigation(sceneIndex);

  // 步骤 4: 组装注入文本
  return {
    prependContext:       // L1 记忆 → 放进用户消息前面
      `<relevant-memories>${memoryLines.join("\n")}</relevant-memories>`,
    appendSystemContext:  // L3 画像 + L2 导航 → 放进系统提示词后面
      `<user-persona>${personaContent}</user-persona>\n\n` +
      `<scene-navigation>${sceneNavigation}</scene-navigation>`,
  };
}
```

### 为什么要分 prependContext 和 appendSystemContext？(行 186-218)

这是为了利用 LLM 提供商的 **Prompt Caching** 能力。

```
appendSystemContext (画像 + 场景导航)
  → 变化频率低（几天变一次）
  → 放进系统提示词的末尾
  → 下一轮对话如果这部分没变 → 从缓存读，省钱

prependContext (L1 记忆)
  → 每轮都不同
  → 放进用户消息的开头
  → 不破坏系统提示词的缓存
```

### hybrid 搜索 (行 511-646)

```typescript
async function searchHybrid(userText, ...) {
  // 并行跑两种搜索
  const [keywordResult, embeddingResult] = await Promise.all([
    // 关键词：FTS5
    buildFtsQuery(userText) → searchL1Fts(),
    // 语义：embedding → vec0 KNN
    embeddingService.embed(userText) → searchL1Vector(),
  ]);

  // RRF 融合：两个排名列表合并
  const mergedMap = new Map();
  for (const r of keywordResults) {
    mergedMap.set(r.id, { rrfScore: 1 / (60 + rank + 1) });
  }
  for (const r of embeddingResults) {
    // 如果同一个记录两边都有 → rrfScore 累加
    mergedMap.set(r.id, { rrfScore: existing + 1 / (60 + rank + 1) });
  }
  // 按 rrfScore 降序排列
  return sorted.slice(0, maxResults);
}
```

## 文件 4：`auto-capture.ts` — 对话录制

> 源码：[src/core/hooks/auto-capture.ts](../src/core/hooks/auto-capture.ts) | 343 行

### 这个文件干什么

每次 Agent 回复结束后被调用，做三件事：

1. **录 L0**：把对话消息写入 JSONL + SQLite
2. **建 L0 向量索引**：如果有 embedding 服务，给 L0 消息算向量
3. **通知管道**：告诉 pipeline-manager "有新对话"，让它判断是否该触发 L1

### 整体入口 (行 45-343)

```typescript
export async function performAutoCapture(params) {
  // Step 1+2: L0 录制 + checkpoint（原子操作）
  await checkpoint.captureAtomically(sessionKey, async (afterTimestamp) => {
    filteredMessages = await recordConversation({
      messages, sessionKey, baseDir: pluginDataDir, afterTimestamp
    });
  });

  // Step 1.5: L0 向量索引
  //   路径 A (SQLite)：先写元数据，后台异步算 embedding
  //   路径 B (TCVDB)：同步算 embedding，一起写入
  for (const msg of filteredMessages) {
    if (supportsBgEmbed) {
      // 路径 A: 只写元数据 → embedding 后台补
      await vectorStore.upsertL0(record, undefined);
    } else {
      // 路径 B: 同步 embedding → 一起写
      const embedding = await embeddingService.embed(msg.content);
      await vectorStore.upsertL0(record, embedding);
    }
  }

  // Step 3: 通知管道
  await scheduler.notifyConversation(sessionKey, []);
}
```

### 为什么分路径 A 和路径 B？(行 147-156)

```
路径 A (SQLite, supportsDeferredEmbedding=true):
  upsertL0(元数据) → 秒级返回 → 用户得到回复快
  embedding 在后台偷偷算 → 几秒后补上去

路径 B (TCVDB, supportsDeferredEmbedding=false):
  必须先算 embedding 才能写 → 同步等待
  因为 VDB 远端存储需要 embedding+元数据一起提交
```

设计要点：**不要让用户等着你的记忆系统**。录对话应该很快返回，不能让 LLM 回复完了还要等 3 秒的 embedding 计算。

### checkpoint 原子操作 (行 105-138)

```typescript
await checkpoint.captureAtomically(sessionKey, pluginStartTimestamp,
  async (afterTimestamp) => {
    filteredMessages = await recordConversation({ afterTimestamp });
    // 返回当前时间戳作为新的 checkpoint
    return { maxTimestamp, messageCount };
  }
);
```

`captureAtomically` 保证：**读 checkpoint → 录新对话 → 更新 checkpoint** 这三步在文件锁的保护下完成。防止两个并发的 `agent_end` 都读到旧的 checkpoint，录出重复的 L0。

## 总结：四个文件怎么配合

```
用户说话
  │
  ▼
index.ts  before_prompt_build 钩子
  │
  ├─→ TdaiCore.handleBeforeRecall()
  │     └─→ auto-recall.performAutoRecall()
  │           ├─ searchMemories() → FTS5/Vector/Hybrid
  │           ├─ 读 persona.md (L3)
  │           └─ generateSceneNavigation (L2)
  │           └─ 返回 prependContext + appendSystemContext
  │
  ▼
LLM 回复
  │
  ▼
index.ts  agent_end 钩子
  │
  ├─→ TdaiCore.handleTurnCommitted()
  │     └─→ auto-capture.performAutoCapture()
  │           ├─ recordConversation() → JSONL + SQLite (L0)
  │           ├─ upsertL0() → vector index
  │           └─ scheduler.notifyConversation() → 管道计数 +1
  │
  ▼
pipeline-manager 判断条件满足
  ├─→ L1 提取（LLM）
  ├─→ L2 场景（LLM）
  └─→ L3 画像（LLM）
```
