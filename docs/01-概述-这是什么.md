# 01 — TencentDB Agent Memory 是什么

> 阅读时间：~10 分钟 | 零门槛

---

## 一句话说清楚

**这是一个给 AI Agent 装"记忆系统"的插件。**

就像人有短期记忆和长期记忆，这个插件让 AI 也能：
- 记住你们之前的对话（短期记忆）
- 从中提取出关于你的偏好、习惯、背景（长期记忆）
- 下次对话时自动调出来用

---

## 为什么需要它？

### 没有记忆的 AI 是什么样

```
你: 我平时用 TypeScript，不喜欢 Python
AI: 好的，记住了！

（第二天）
你: 帮我写个工具函数
AI: 好的！我帮你用 Python 写...  ← 完全忘了昨天的对话
```

每次对话都是"初次见面"，你得反复告诉它同样的事情。

### 有记忆的 AI 是什么样

```
你: 帮我写个工具函数
AI: 好的！根据你之前的偏好，我用 TypeScript 来实现...  ← 想起来了
```

## 它到底做了什么事？——一张图看懂

```
每次你和 AI 说话
      │
      ▼
┌─────────────────────────────┐
│         原始对话              │  ← 这层叫 L0
│   "我平时用 TypeScript，       │     原封不动保存下来
│    不喜欢 Python"              │     存在 conversations/ 目录
└─────────────┬───────────────┘
              │  LLM 自动提取
              ▼
┌─────────────────────────────┐
│        结构化事实              │  ← 这层叫 L1
│   类型: 偏好                   │     每条是"一条完整的信息"
│   内容: 用户使用 TypeScript     │     存在向量数据库里，可搜索
│   类型: 偏好                   │
│   内容: 用户不喜欢 Python       │
└─────────────┬───────────────┘
              │  LLM 自动归纳
              ▼
┌─────────────────────────────┐
│         场景块                 │  ← 这层叫 L2
│   场景: 编程语言偏好            │     相关事实归为一类
│   - 主语言: TypeScript         │     存为 Markdown 文件
│   - 避免: Python               │     人类也可以直接阅读
└─────────────┬───────────────┘
              │  LLM 自动总结
              ▼
┌─────────────────────────────┐
│         用户画像               │  ← 这层叫 L3
│   用户是一个 TypeScript 开发者   │     最抽象、最稳定的描述
│   偏好静态类型、现代工具链        │     每次对话都会加载
│   ...                         │
└─────────────────────────────┘
```

这就是项目的核心思路——**记忆不应该是扁平的一堆碎片，而应该是层层提炼的金字塔**。

---

## 核心概念：L0 → L1 → L2 → L3

| 层级 | 名字 | 是什么 | 存在哪 | 谁来产生 |
|------|------|--------|--------|----------|
| **L0** | 原始对话 | 你和 AI 的每句话，原封不动 | `conversations/` 目录下的 JSONL 文件 | 自动录制，不经过 LLM |
| **L1** | 结构化事实 | 从对话中提取的一条条"信息原子"，如"用户叫小明""小明喜欢 TypeScript" | SQLite 数据库（vectors.db）的 `l1_records` 表 | LLM 从 L0 对话中提取 |
| **L2** | 场景块 | 把相关的 L1 事实归到一类，形成 Markdown 文件，如"编程偏好""旅行计划" | `scene_blocks/` 目录下的 .md 文件 | LLM 从 L1 记忆中归纳 |
| **L3** | 用户画像 | 最抽象的一层，描述用户是谁、什么习惯、什么风格 | `scene_blocks/persona.md` | LLM 从 L2 场景块中总结 |

### 为什么分四层？而不是直接把对话全存了？

```
如果只存原始对话（只有 L0）:
  问题: 对话太多，Token 爆炸，AI 看不过来
  你遇到过吧？聊天记录翻到三页前就找不到了

如果把所有对话让 LLM 总结成一段（L0→L3 一步跳）:
  问题: 总结是不可逆的！"用户喜欢 TypeScript"被总结进去了，
        但如果总结错了，原始证据丢了，永远无法查证

四层方案:
  每一层都能追溯到下一层，就像论文的引用链：
  L3 说用户喜欢 TypeScript →
    翻 L2 的编程偏好场景 →
      翻 L1 的结构化事实 →
        翻 L0 的原始对话："我平时用 TypeScript"
  证据永远不丢，但 AI 不用每次都从头看
```

## 两个场景：一条消息进入系统后发生什么

### 场景一：说话时 → 存入记忆（捕获）

```
你发了一条消息 "帮我查一下东京的天气预报"
      │
      ▼
┌──────────┐
│ L0 录制    │  原封不动存入 conversations/2026-06-30.jsonl
└─────┬────┘
      │  （积累了 5 轮对话后）
      ▼
┌──────────┐
│ L1 提取    │  LLM 看最近 5 轮对话，提取事实：
└─────┬────┘  "用户正在计划一次东京旅行"
      │
      │  （积累了更多相关 L1 记忆后）
      ▼
┌──────────┐
│ L2 场景    │  LLM 归纳："旅行计划：用户要去东京，时间在六月"
└─────┬────┘  存入 scene_blocks/travel-tokyo.md
      │
      │  （积累了 50 条新 L1 记忆后）
      ▼
┌──────────┐
│ L3 画像    │  LLM 更新 persona.md，加入"用户喜欢旅行"等长期特征
└──────────┘
```

### 场景二：下次对话时 → 自动调出（召回）

```
你发了一条 "上次说去东京，找个酒店"
      │
      ▼
┌──────────────┐
│ 召回 L3 画像   │  "这个用户是旅行爱好者，偏好..."
│（放入系统提示） │  每次都加载，像背景信息
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 召回 L1 事实   │  搜索到："用户计划去东京旅行""旅行时间在六月"
│（放入用户提示） │  这次对话相关的记忆
└──────┬───────┘
       │
       ▼
   AI 看到你的消息 + 记忆上下文 →
   "根据你之前的计划，推荐东京新宿附近的酒店..."
```

## 这个项目不是什么

- ❌ 不是远程云服务——所有数据存在你本地
- ❌ 不是 AI 模型——它不会代替 LLM，它"辅助"LLM
- ❌ 不是聊天应用——它是插件，依附在 OpenClaw/Hermes 这样的 Agent 框架上
- ❌ 不是全文存档搜索——它是有结构的记忆，不是 Ctrl+F
