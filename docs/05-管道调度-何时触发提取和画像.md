# 05 — 管道调度：何时触发提取和画像

> 阅读时间：~15 分钟 | 理解触发条件即可，不需要全懂代码

---

## 问题：为什么需要调度？

L1/L2/L3 每一层都要调 LLM。LLM 调用**又贵又慢**——调一次可能花几秒钟、几毛钱。所以不能每说一句话就跑。

需要一套规则来决定：**什么时候值得跑一次 LLM 提取/归纳/画像？**

这就是 PipelineManager 的职责。

---

## 三层触发的总览

```
                     L1 提取             L2 场景             L3 画像
                     ────────            ────────            ────────
触发条件 1         每 5 轮对话          L1 跑完后          每 50 条新 L1
触发条件 2         10 分钟空闲          延迟 10 秒
触发条件 3                             最少间隔 15 分钟
                                      最长间隔 1 小时
```

---

## L1 触发：三种条件，满足其一就跑

### 条件 1：轮次计数器

每轮对话结束后，计数器 +1。到了阈值（默认 5）就触发。

```typescript
// pipeline-manager.ts（简化）
everyNConversations = 5;  // 默认值，可配置

notifyTurnCommitted(sessionKey) {
  const count = (this.rounds.get(sessionKey) || 0) + 1;
  this.rounds.set(sessionKey, count);

  if (count >= this.everyNConversations) {
    this.scheduleL1(sessionKey); // 触发 L1！
    this.rounds.set(sessionKey, 0); // 重置计数器
  }
}
```

### 条件 2：空闲超时

如果用户说了 3 轮就停了（不够 5 轮），计时器启动。10 分钟后还没新消息 → 自动触发。

```typescript
// 简化
idleTimeout = 600; // 秒，默认 10 分钟

notifyTurnCommitted(sessionKey) {
  // 每次有对话就重置计时器
  clearTimeout(this.idleTimers.get(sessionKey));
  this.idleTimers.set(sessionKey, setTimeout(() => {
    this.scheduleL1(sessionKey); // 空闲超时 → 触发！
  }, this.idleTimeout * 1000));
}
```

### 条件 3：防重入

如果上一轮 L1 还没跑完，新的不会启动。用 `isPipeliningRunning` 标记控制。

### warmup 是什么

默认开启 `enableWarmup`。意思是**刚开始用的时候更积极地触发**：
- 第 1 次：1 轮就触发
- 第 2 次：2 轮就触发
- 第 3 次：3 轮触发
- ...
- 第 N 次及以后：N 轮触发（回到正常频率）

目的：新用户刚装插件时，快速建立起初始记忆，不要等很久才第一次生效。

## L2 触发：L1 之后延迟归纳

L2 的触发条件是 **"L1 刚刚跑完"**，但有节流：

```typescript
// pipeline-manager.ts（简化）
l2Delay = 10;           // L1 跑完等 10 秒再跑 L2
l2MinInterval = 900;    // 两次 L2 之间最少隔 15 分钟
l2MaxInterval = 3600;   // 即使条件不满足，1 小时也强制跑一次

scheduleL1(sessionKey) {
  // ... 跑 L1 ...
  // L1 完成后：
  if (now - lastL2Time >= l2MinInterval) {
    // 间隔够了 → 延迟 10 秒触发 L2
    setTimeout(() => this.scheduleL2(sessionKey), l2Delay * 1000);
  } else {
    // 间隔不够 → 设置一个定时器，到最大间隔时强制触发
    setTimeout(() => this.scheduleL2(sessionKey), l2MaxInterval * 1000);
  }
}
```

### 为什么 L2 要等？

场景归纳是"总结一批 L1 事实"，如果 L1 才出来 1-2 条就归纳，场景会很碎。等够间隔，积攒一批相关事实再做归纳，场景更有意义。

## L3 触发：积攒多了才更新

```typescript
triggerEveryN = 50;  // 每 50 条新 L1 记忆触发一次

scheduleL3(sessionKey) {
  const newMemoriesSinceLastPersona = this.countNewL1SinceLastL3(sessionKey);
  if (newMemoriesSinceLastPersona >= this.triggerEveryN) {
    // 凑够了 50 条 → 更新画像
    this.runPersonaGeneration(sessionKey);
  }
}
```

### 为什么 L3 最少触发？

用户画像应该是**稳定的**。一个人的核心偏好不会一天变三次。50 条新事实才触发一次，确保画像反映的是长期模式，不是临时波动。

## 管道状态持久化

**代码位置**：[src/utils/checkpoint.ts](../src/utils/checkpoint.ts)

### 问题

OpenClaw 可能会重启。重启后如果计数器都丢了，又得重新积累 5 轮才触发 L1。

### 解决方案

PipelineManager 把状态存到磁盘：

```json
// ~/.openclaw/state/memory-tdai/pipeline-checkpoint.json
{
  "l1RoundCounts": { "wechat:user_123": 3 },
  "lastL1Time": "2026-06-30T10:00:00Z",
  "lastL2Time": "2026-06-30T09:45:00Z",
  "newL1CountSinceLastPersona": 32
}
```

重启后读回来，计数器接着用。

## Session 生命周期

**代码位置**：[src/core/tdai-core.ts](../src/core/tdai-core.ts)

### 什么是 Session

一个对话会话。在 OpenClaw 里，每个 `/new` 创建的新对话就是一个 session。

### Session 结束时

```typescript
// tdai-core.ts
handleSessionEnd(sessionKey) {
  // 1. 清掉该 session 的空闲计时器
  // 2. 把该 session 的计数器归零
  // 3. 但不影响其他 session
}
```

### 全局关闭时

```typescript
// tdai-core.ts
async destroy() {
  // 1. 停止所有计时器
  // 2. 等待正在跑的 L1/L2/L3 任务完成
  // 3. 持久化状态到磁盘
  // 4. 关闭 SQLite 连接
  // 5. 关闭 embedding 服务
}
```

这个顺序很重要——如果先关 SQLite 再等 L1 完成，L1 就写不进去了。

## 常见问题

### Q: 为什么装了插件后第一次对话没有记忆？

A: 默认 `everyNConversations: 5`，需要说够 5 轮才会第一次触发 L1 提取。如果开了 `enableWarmup`，第一次 1 轮就会触发。

### Q: 为什么画像一直不更新？

A: 默认 `triggerEveryN: 50`，需要积累 50 条新 L1 事实才会触发画像更新。刚开用的话可能要等几天。

### Q: 怎么让它更频繁地触发？

A: 调小 `everyNConversations`（比如 2）和 `triggerEveryN`（比如 20）。但确认你的 LLM 预算——每次 L1/L2/L3 都要花 token。
