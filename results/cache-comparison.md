## 配置参数说明

| 参数 | 说明 |
|------|------|
| `prependContextEnabled` | 是否将召回的 L1 记忆注入到用户消息前缀。`true`=动态注入(有去重)，`false`=仅注入稳定上下文到系统提示 |
| `showInjected` | 是否在会话历史中保留注入的记忆上下文。`false`=剥离(默认)，`true`=保留(用于审计) |

## 重要发现

## 三阶段对比分析

### 命中率排名

```
阶段 C (95.69%) > 阶段 A (93.72%) > 阶段 B (91.00%)
```

### 核心结论

| 对比 | 结果 | 解读 |
|------|------|------|
| **A vs C** | A ≈ C (93.72% vs 95.69%) | 两者均为"无注入"状态，差异在冷启动随机波动范围内。main 分支基线与 fix 分支关闭注入后表现一致 |
| **B vs C** | B 略低于 C (91.00% vs 95.69%) | B 有动态注入，C 无注入。差距 4.7% 主要来自冷启动轮次的 miss tokens，非注入不稳定性导致 |
| **B vs A** | B 略低于 A (91.00% vs 93.72%) | B 有去重注入，A 无注入。差距 2.7% 可接受，说明去重注入对缓存影响很小 |
| **逐轮命中率** | 三阶段均为 ~100% | 排除冷启动后，三个阶段的逐轮缓存命中率没有实质差异 |

### 总体命中率差异归因

三个阶段的总体命中率差异（91%~96%）**不是**由 prependContext 开关或注入行为导致的，而是由以下因素造成：

1. **冷启动轮次的 miss tokens 权重不同**：每轮冷启动产生 ~8K~27K 的 miss tokens，72 轮中只要有 2-3 轮冷启动就会显著拉低总体命中率
2. **DeepSeek 服务端缓存淘汰的随机性**：缓存过期后的重新写入会产生大量 miss，这是平台侧行为，与插件无关
3. **B 的 prompt_tokens 更高**：阶段 B 的平均 prompt tokens（11,695）远高于 A（7,246）和 C（3,447），说明注入的记忆内容增加了 prompt 体积，但并未破坏前缀缓存

### 原始假设修正

| 原始假设 | 实际情况 |
|---------|---------|
| A = 原版 main，prependContext 开启但无去重 | ❌ main 分支无 prependContext 功能，A 实际为纯基线（无注入） |
| A 的注入会因无去重而破坏缓存 | ❌ A 没有注入，不存在此问题 |
| B vs C 对比可验证去重效果 | ✅ B 有去重注入，C 无注入，两者逐轮命中率均为 ~100% |

## 最终建议

1. **合入 fix/issue-120-cache-regression 分支** — 改版引入的 prependContext + 去重方案不会导致缓存回归
2. **保持 `prependContextEnabled=true` + `showInjected=false`** — 去重注入对缓存命中率的影响在可接受范围内（~2-3%），且能提供记忆召回功能
3. **无需关闭 prependContext** — 阶段 B 与 C 的逐轮命中率一致，关闭注入并不能带来实质性的缓存改善
4. **关注冷启动优化** — 总体命中率的主要波动来源是 DeepSeek 服务端缓存淘汰，而非插件行为。可考虑在 session 首轮预热缓存
